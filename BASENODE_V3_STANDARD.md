Архитектурный Стандарт Ноды V3: «Event-Data Bus»
 1.Каждая нода в системе должна реализовывать унифицированный интерфейс, состоящий из фиксированного набора входных и выходных портов (пинов). Это обеспечивает предсказуемость исполнения и упрощает каскадную обработку данных.
   1.1.Входные порты (Input Ports)
       1.1.1.Trigger In (Сигнал управления): Инициирует выполнение логики ноды. Нода не начинает работу, пока не получит сигнал по этому порту.
       1.1.2. Data In (Шина данных): Принимает легковесный ссылочный пакет (DataBusPacket): {SessionID, DataID, Type, Metadata}. Сами данные (Content) по проводам НЕ передаются. Нода самостоятельно запрашивает тяжелую полезную нагрузку из центральной Шины (DataBus), используя композитный ключ "{SessionID}_{DataID}".
   1.2.Выходные порты (Output Ports)
       1.2.1.Success Trigger (Сигнал успеха): Активируется после успешного выполнения основного действия. Запускает следующую ноду в основной цепочке.
       1.2.2.Data Out (Поток данных): Передает основной результат обработки (напр., текст или объект).
       1.2.3.Custom Data Out (Специфический поток): Настраиваемый выход для передачи метаданных или фрагментов данных. Настраивается внутри ноды (какое поле из {Metadata} выводить).
       1.2.4.Error/Rejected Trigger (Поток спасения): Активируется в случае сбоя или невалидных данных. Запускает альтернативную цепочку (ветку «План Б»).
       1.2.5. Black Box Log (Поток телеметрии и визуальной отладки): Выход для технической диагностики, работающий в гибридном режиме (Dual-Mode):
             1.2.5.1. Фоновый режим (Всегда активен): Нода мгновенно и без блокировок (через Channel<T>.TryWrite) отправляет системный лог в in-memory буфер. Это защищает систему от фризов при записи на диск.
             1.2.5.2. Визуальный режим (По требованию): Порт физически присутствует на холсте ноды. Пользователь может протянуть от него провод к другим нодам (например, "Overlay Text" или "Telegram Alert"). При генерации лога нода формирует DataBusPacket и отправляет его по этому проводу. Это позволяет выводить логи на экран в реальном времени для отладки. Если провод не подключен — передача пакета безопасно игнорируется.
   1.3.Специализация и наследование (GuardianNode Compliance)
       1.3.1.Базовая совместимость: Любая нода, выполняющая роль «Стража» (GuardianNode), обязана строго соблюдать интерфейс портов BaseNode.
       1.3.2.Дополнительные обязательства: В дополнение к стандартному жизненному циклу, GuardianNode должна реализовывать:
             1.3.2.1.Контроль целостности сессий (SessionID Validation).
             1.3.2.2.Атомарную запись в Black Box Log перед активацией любого выходного триггера.
             1.3.2.3.Внутренний системный таймаут (Watchdog), работающий независимо от основного цикла обработки данных, для предотвращения зависания потока.
       1.3.3.Приоритет ошибки: При конфликте между логикой BaseNode и спецификой GuardianNode (например, при критическом сбое), приоритет всегда отдается протоколу обработки ошибок GuardianNode с последующей записью в Black Box.
   1.4.Стандарт передачи данных (DataBus Packet Protocol)
       1.4.1.Структура пакета: Любые данные, передаваемые через порты Data Out или Custom Data Out, обязаны быть инкапсулированы в объект DataBusPacket:
             1.4.1.1. Стандартная информация (System Payload): {SessionID, DataID, Timestamp, SourceNodeID, Status}. Этот пакет курсирует по основному порту Data Out.
             1.4.1.2. Динамическая информация (Content / Context Payload): Любые специфические данные (текст, байты аудио, JSON). Физически хранятся строго в DataBus. По проводу Custom Data Out передается только расширенный набор Metadata и дополнительный DataID (если нода генерирует вторичный поток данных).
             1.4.1.3. Унифицированные метаданные (Metadata Object): Любой DataBusPacket обязан включать объект {Metadata}, который является «паспортом» данных. Ноды имеют право считывать из него системную информацию и дописывать свои параметры. Все уникальные ключи метаданных, используемые нодами, должны быть согласованы с общим реестром в system_map.json. Это позволяет нодам «понимать» контекст данных (например, уровень доверия фразы или ID ассистента) без необходимости парсить основной контент.
             1.4.1.4. Типизация метаданных (Tagging Standard): Все метаданные, передаваемые в строковом виде внутри DataBusPacket, должны быть обернуты в типизированные тэги формата {Type:Value}. Это исключает двусмысленность и позволяет нодам фильтровать только те данные, которые им необходимы для работы.
             1.4.1.5. Механизм подписки на метаданные: Каждая нода обязана реализовать «Реестр подписок» в своих настройках. При обработке DataBusPacket нода взаимодействует только с теми типами метаданных, на которые она явно подписана. Данные, не входящие в список подписки, проходят через ноду транзитом без изменений, обеспечивая целостность сквозного контекста.
       1.4.2.Правило атомарности: Активация порта Success Trigger является сигналом для следующей ноды, что данные в Data Out и Custom Data Out прошли валидацию и готовы к чтению.
       1.4.3.Обязательство ноды: Каждая нода при реализации обязана определить, какая часть её вывода является Стандартной, а какая — Динамической, и направить их в соответствующие порты.
 2.Правила внутренней обработки (Логика ноды)
   2.1.Типизация и фильтрация: На входе Data In нода проверяет тип пакета. Если нода не может обработать текущий тип данных, она обязана передать его дальше без изменений через порты Data Out или Error Trigger (прозрачное прохождение).
   2.2.Изоляция: Любая критическая ошибка, возникшая внутри ProcessData(), должна приводить к вызову метода HandleError(). Этот метод автоматически упаковывает детали ошибки в лог и посылает импульс в Error Trigger.
   2.3.Автономность портов: Если порты не задействованы в макросе (не соединены проводами), нода просто игнорирует передачу сигналов в эти порты, не нарушая стабильность системы.
   2.4.Единый контракт: Любая новая нода наследует этот набор портов. Нейросеть-кодер при создании новой ноды обязана следовать этой структуре.
   2.5.Протокол долгих операций (Long-Running Tasks):
       2.5.1.Для узлов, предполагающих длительное выполнение (PowerShell, загрузка файлов, ожидание окна), применяется защита от ложного срабатывания Watchdog.
             2.5.1.1.Настраиваемый лимит: Такие ноды обязаны иметь в Settings параметр NodeTimeoutMs (0 = бесконечность), который переопределяет глобальный таймаут планировщика.
             2.5.1.2.Динамический Keep-Alive: Если процесс ноды генерирует поток данных (например, вывод в консоль stdout), каждое получение порции данных обязано вызывать внутренний метод ResetWatchdogTimer(). Это сообщает планировщику, что поток активен, даже если общее время выполнения превысило базовые лимиты.



Протокол взаимодействия метаданных (V3.4)
Система базируется на Интерактивном контракте, где данные передаются «умными пакетами», а их структура определяется физическими связями между нодами.
1. Структура данных (DataBusPacket)
Любой пакет, передаваемый через Data-порты, включает:
System Payload: {SessionID, Timestamp, SourceNodeID, Status}.
Context Payload: Основные данные (текст, байты, JSON).
Metadata Object: «Паспорт» данных в формате типизированных тэгов {Type:Value}. Это обеспечивает строгую типизацию и предотвращает конфликты имен.
2. Интерактивный контракт (Metadata Discovery)
Визуализация: На каждой ноде расположена кнопка вызова списка метаданных.
При нажатии на нее отображаются два списка:
Generated by Node: Метаданные, которые нода создает сама.
Inherited from Upstream: Наследуемые метаданные от всех подключенных к ней нод по «трассе» данных.
Физическая зависимость: Список Inherited активируется только при наличии физического соединения провода передачи данных (Data Out -> Data In).
Без провода нода видит только собственный список.
Однонаправленность: Контекст движется строго Downstream (по направлению сигнала).
Нода не может видеть метаданные «будущих» узлов.
3. Управление контекстом (Subscription & Isolation)
Фильтрация: В настройках ноды (раздел «Metadata Subscription») пользователь выбирает только те типы метаданных, которые необходимы для работы в текущем сценарии.
Изоляция: Если нода не подписана на метаданные, она их игнорирует, передавая транзитом.
Если нода пытается обратиться к несуществующим метаданным — система блокирует связь (визуальная индикация ошибки).
Внешние данные: Для данных из интернета или ОС используется отдельная «Passive Configuration Node» (без портов Trigger), которая выступает глобальным поставщиком данных для всего макроса.
4. Принцип визуальной отладки
Система автоматически подсвечивает путь данных.
Если провод разорван, визуальная цепь метаданных прерывается, исключая «магические» сбои.
Пользователь всегда точно знает, какие данные «текут» через каждый конкретный провод в каждый момент времени.



Архитектурный стандарт: Drag-and-Drop Mapping (V3.6)
1. «Умные поля» (Smart Fields)
Визуальный отклик: Поля, поддерживающие метаданные (например, «Клавиша», «Задержка»), при наведении на них чипа с метаданными из нижнего списка (область под «Тест цепочки») слегка подсвечиваются или меняют цвет рамки.
Типовая фильтрация: Если ты тянешь «числовой» тэг (например, {Sys:CPU_Load}) к текстовому полю, оно не подсвечивается (защита от дурака).
Состояние по умолчанию: Если метаданные не перетащены, поле работает в статичном режиме (ты сам вводишь 50 мс). Как только ты «бросил» метаданные в поле, оно переключается в режим Dynamic Input.
2. Интерфейс управления маппингом (Нижний блок)
Вместо того чтобы захламлять каждое поле мелкими кнопками, мы используем единый нижний блок (область, выделенная на твоем скриншоте image_573ca8.png красным):
Список активных связей: В этой зоне после перетаскивания автоматически создается строка вида:
[Поле: Задержка] : {Sys:CPU_Load} [X]
[Поле: Клавиша] : {Input:DetectedKey} [X]
Крестик (Удаление): Нажатие на [X] моментально разрывает связь, и поле возвращается к статичному значению.
Визуальная чистота: Ты видишь весь список активных метаданных для этой ноды в одном месте, не лазая по каждой строчке настроек.
3. Логика исполнения
Иерархия приоритетов:
Если в поле перетащены метаданные — используем их.
Если поле пустое или метаданные удалены — используем статичное значение из самого поля.
Блокировка: Как ты и предложил, если чекбокс «Случайная задержка» выключен, система программно блокирует «прием» метаданных для этого поля (оно не подсвечивается при наведении), чтобы пользователь не мог настроить то, что отключено.



Протокол потоков данных (V3.7)
1. Разделение магистралей
DataOut (Основной провод): Передает «тело» макроса — команды, текст, числовые значения, нажатия клавиш. Это обязательная магистраль для выполнения действий.
Custom DataOut (Провод контекста): Передает «паспорт» данных — метаданные {Type:Value}. Это «умная» магистраль, которая несет исключительно контекст и параметры.
2. Динамика списков метаданных
При подключении DataOut:
Нода получает «базовый» контекст (системный статус, базовые параметры выполнения).
Список метаданных в ноде пополняется данными, которые несет именно этот поток.
При подключении Custom DataOut:
Нода делает «расширенное обновление» списка.
Сюда добавляются только те специфические метаданные, которые приходят из внешней среды или от специальных нод-источников (Passive Configuration).
Важно: Этот провод работает как «добавка». Он не дублирует данные основного провода, а дополняет общую картину.
3. Объединение в ноде (The Merge Point)
Внутри ноды оба потока встречаются. Нода собирает «Полный Контекст» (Data + Custom Data).
Теперь, когда ты открываешь список метаданных в ноде, ты видишь комбинированный список: то, что пришло по основному проводу, плюс то, что пришло по проводу Custom Data.



Архитектурный Стандарт v3: «Unified Bus Architecture»
 1.Каждая нода в системе является автономным модулем, взаимодействующим с общей «Шиной данных» (Data Bus). Шина разделяет поток управления (когда делать) и поток данных (с чем делать).
   1.1.Порты ввода/вывода (Интерфейс)
       1.1.1.Trigger In: Порт управления. Ожидает импульс для старта выполнения логики.
       1.1.2.Data In: Порт приема ссылки на пакет данных из Шины (DataID).
       1.1.3.Success Trigger: Импульс, передаваемый следующей ноде в цепочке после завершения работы.
       1.1.4.Data Out: Передает DataID результата обработки.
       1.1.5.Error Trigger: Импульс для запуска альтернативной цепочки в случае сбоя.
       1.1.6.Black Box Log: Постоянный поток телеметрии (JSON-объект) в лог-менеджер.
   1.2.Система хранения данных (Data Bus)
       1.2.1.Шина — это центральное хранилище данных, к которому обращаются ноды.
       1.2.3.Уровень «Конфигурация» (Постоянные данные):
       1.2.3.Хранятся внутри ноды (Node.Settings).
       1.2.4.Задаются в редакторе (статичны).
       1.2.5.Пример: Имя триггера, пороги срабатывания.
       1.2.6.Уровень «Контекст» (Временные данные):
       1.2.7.Хранятся в Шине (DataBus).
       1.2.8.Доступны по уникальному DataID.
       1.2.9.Живут от триггера до окончания работы макроса.
       1.2.10.Динамическая перезапись: Нода имеет право принудительно обновить свои Settings на основе данных из DataBus (механизм Hot-Swapping), если это заложено в логику.
   1.3.Правила «Прозрачного прохождения» (Data Flow)
       1.3.1.Изоляция данных и Fail-Fast Sessioning: Нода не копирует данные, а работает по ссылке на Шину (DataBus).
             1.3.1.1.При многопоточном выполнении Шина Данных хранит переменные с использованием композитного ключа в формате: "{SessionID}_{DataID}".
             1.3.1.2.Это снимает часть ответственности с самой ноды. Если нода попытается обратиться к DataBus со своим устаревшим или чужим SessionID, шина прозрачно вернет null, вызвав естественный сбой на уровне интерфейса без наслоения переменных из параллельных потоков.
       1.3.2.Прозрачность: Если нода не умеет обрабатывать пришедший тип данных (напр. «Ошибку»), она не уничтожает его, а передает дальше по порту Data Out. Информация должна дойти до ноды-обработчика (напр. логгера или оверлея).
       1.3.3.Валидация: Нода проверяет данные по типу: {Content, Type, Metadata}.
 2.Внедрение «Контроллера Сессий» в BaseNode
   Каждая нода (будь то «Действие», «Условие» или «Синхронизатор») при получении Trigger In должна выполнять Pre-flight check (предварительную проверку):
   2.1.Этап А (Валидация): Нода считывает SessionID из входящего пакета.
   2.2.Этап Б (Сравнение): Если SessionID новый — нода обновляет свой внутренний регистр ActiveSessionID и сбрасывает свои внутренние счетчики. Если SessionID старый (или отсутствует) — сигнал немедленно отправляется в Rejected Trigger с кодом ExpiredSession.
   2.3.Этап В (Атомарность): Только после прохождения проверки нода переходит к исполнению полезной нагрузки.


Системный контракт (Interface to System Map):
 1.Любая нода (или главный контроллер макроса) при инициализации обязана считать Context Binding из файла {ID}.json.
 2.При получении сигнала Success Trigger в цепочке макроса, контроллер обязан сверяться с RegionID (из системной карты), чтобы понимать, должен ли он отправить макрос в очередь или может выполнять его мгновенно.



Стандарт «Состояния ноды» (Node Lifecycle States)
 1.Сейчас у нас есть «вход» и «выход». Но как нода «понимает», что она делает прямо сейчас? Нужен стандарт состояний:
   1.1.IDLE (Ожидание) — нода спит, потребление ресурсов 0.
   1.2.PROCESSING (Работа) — нода активна (например, идет OCR).
   1.3.WAITING (Ожидание данных) — нода встала на паузу, потому что шина еще не ответила.
   1.4.ERROR (Ошибка) — нода «упала», ждет сигнала рестарта.
   1.5.Зачем это нужно: Чтобы ты мог в оверлее или редакторе подсвечивать ноды. Если нода горит красным (ERROR), ты сразу знаешь, куда тыкать, а не гадаешь, какой макрос завис.



Стандарт «Типизированных портов» (Port Connection Rules)
 1.Типы данных (Data Types)
   1.1.Все порты в программе делятся на группы. Соединение возможно только если типы портов совпадают:
       1.1.1.SIGNAL: Для проводов типа Trigger (Success, Error, Abort).
       1.1.2.TEXT: Для передачи строковых данных.
       1.1.3.IMAGE: Для передачи скриншотов или графических буферов.
       1.1.4.OBJECT: Для сложных структур данных (например, конфиги или списки).
       1.1.5.ANY: «Универсальный порт» (используй крайне редко, только для очень гибких нод-фильтров).
 2.Механика «Магнитного притяжения» (Connection Filtering)
   2.1.Порт-источник (Выход): «Знает», какой тип данных он генерирует.
   2.2.Порт-приемник (Вход): «Знает», какой тип данных он ожидает.
Логика: В момент, когда ты берешь конец провода, редактор подсвечивает (или активирует) только те входы, которые имеют такой же тип данных. Остальные входы для провода «невидимы» — они не «магнитятся», и провод к ним просто не прицепится.



Стандарт «Безопасного прерывания» (Abort/Cancel)
 1.Что если макрос зациклился или пользователь передумал?
   1.1.Нужен «Сигнал прерывания» (Global Abort Signal): У каждой ноды должен быть встроенный «предохранитель». Если прилетает глобальный сигнал ABORT, нода обязана мгновенно сбросить свои текущие вычисления, очистить временные данные в шине и вернуться в IDLE.

Архитектурный Стандарт: Critical Section Protection (CSP)
 1.Для каждой ноды в системе вводится механизм безопасного прерывания с явным предупреждением об ответственности.
   1.1.UI-элемент (Тумблер)
       1.1.1.В настройках каждой ноды присутствует переключатель: «Игнорировать прерывание (Critical Section)».
       1.1.2.Визуальное предупреждение: Рядом с тумблером находится значок «Внимание» (желтый треугольник). При наведении на него всплывает подсказка:
       1.1.3.«ВНИМАНИЕ: Активация данного параметра запрещает принудительную остановку этой ноды до завершения её текущего действия. Некорректное использование может привести к зависанию цепочки макроса или повреждению данных. Используйте только для критически важных задач (запись файлов, сетевые запросы, изменение системных конфигов).»
   1.2.Программная логика (Backend)
       1.2.1.Атрибут IsCriticalSection: В базовом классе BaseNode поле, которое управляет логикой выполнения.
       1.2.2.Логика обработки AbortSignal:
             C#
             public void OnAbortSignalReceived() {
                 if (!IsCriticalSection) {
                     ExecuteCleanup(); // Мгновенная остановка
                 } else {
                     // Логируем в Black Box, что сигнал получен, но проигнорирован из-за Critical Section
                     LogManager.Write(NodeID, "Abort signal received but ignored: Critical Section active.");
                 }
             }
       1.2.3.Защита от «вечного зависания» (Graceful Shutdown): Принудительное жесткое «убийство» рабочего потока (.NET Thread Kill) категорически запрещено из-за риска повреждения системных процессов (например, драйверов GPU) или памяти.
             1.2.3.1.Механика отмены: Если нода объявляет себя IsCriticalSection, при поступлении Abort ей дается Soft Timeout (например, 5-10 секунд) на корректное достижение безопасной точки сохранения.
             1.2.3.2.Разрешение конфликта: Остановка макроса реализуется только через системный каскадный CancellationToken. Если таймаут превышен, система отменяет Token, принудительно разрывая внутренние I/O ресурсы ноды (сокеты, хэндлы окон) и переводя её в состояние ERROR с отписью в Black Box Log: «Critical Section Soft Timeout exceeded: Context cascade canceled».



СИСТЕМНЫЙ ПРОТОКОЛ: HEARTBEAT (Проверка пульса)
 1.Цель: Детекция «зомби-процессов» (зависших нод), которые не отвечают планировщику.
   1.1.Механика: Глобальный планировщик системы периодически (интервал T) рассылает сигнал Ping всем нодам, находящимся в статусе PROCESSING.
   1.2.Контракт ноды: Каждая нода обязана реализовать метод onPing().
       1.2.1.В ответ нода должна вернуть сигнал Pong, подтверждающий работоспособность потока.
   1.3.Арбитраж: * Если система не получает Pong от ноды 3 раза подряд, она принудительно переводит ноду (и весь связанный с ней макрос) в статус ZOMBIE.
       1.3.1.Действие: Система вызывает принудительную терминацию потока/процесса и логирует ошибку в Black Box.



Система очередей (Регионы)
Каждый «регион» — это отдельный логический контейнер:
Региональный ID: Каждый макрос в метаданных имеет поле region_id.
Файл очереди: /queues/{RegionID}.json — содержит массив ID макросов
Режим по умолчанию: Все макросы, добавленные в регион, выполняются в режиме «живой очереди» (параллельно или по мере поступления).
Настройка приоритета:
Вызывается через контекстное меню (правой кнопкой мыши по макросу во вкладке очередей).
В появившемся окне пользователь вводит число от 1 до 999.
Значение 1 — наивысший приоритет.
Значение 0 — приоритет не задан (макрос выполняется на общих основаниях).
Область действия: Приоритет работает исключительно внутри конкретного региона.
Бэкап: При каждом изменении очереди текущий файл перезаписывается в /queues/backup/, замещая предыдущий файл того же региона.

Привязка процессов (Context Binding)
Настройки для каждого макроса хранятся в его JSON-структуре (в блоке метаданных):
Тип привязки:
GLOBAL: без привязки.
MANUAL: конкретный путь к process.exe.
FOCUS: активация только при фокусе конкретного окна.
Фильтрация: Строковое поле title_filter для точного определения окна (заголовка процесса).
